# Trace — Model Card

Trace runs three machine-learning models. This document describes each one
honestly: what it does, how it was trained, how well it performs, and where
it should not be trusted. The goal is a model card a clinician or judge can
read in five minutes and understand the full ML story.

---

## At a glance

| Model | Architecture | Where it runs | What it produces |
|---|---|---|---|
| **CV head** | MobileNetV3-Large (fine-tuned, 8 classes) | Cloud (FastAPI + PyTorch, optional) | Bite-type classification + Grad-CAM overlay |
| **Fusion head** | Gradient-boosted decision trees (80 stages, depth 3) | On-device (pure-TS tree walker) | 3-way calibrated probability over {no-Lyme, early-Lyme, disseminated-Lyme} |
| **Temporal head** | GRU (1 layer, hidden=16, ~1.5k params) | On-device (pure-TS forward pass) | Per-day "trajectory probability" → Timeline sparkline |

Why three models instead of one big one: each addresses a distinct kind of
evidence (one image, one snapshot, one time series). They are independently
trainable and independently useful, and the heuristic risk engine remains in
place as a non-ML floor.

---

## 1. CV head — bite photo classifier

**Architecture:** MobileNetV3-Large with the final classifier head replaced
by a `Linear(in_features, 8)` layer. Initialized from ImageNet pretrained
weights. Standard ImageNet normalization at inference.

**Training data:** Kaggle "Bug Bite Images" dataset
(`moonfallidk/bug-bite-images`), 8 classes — `ants`, `bed_bugs`, `chiggers`,
`fleas`, `mosquitos`, `no_bites`, `spiders`, `ticks`. Augmentation:
horizontal flip, ±15° rotation, brightness/contrast jitter ±0.2. 80/20
train/val split with `random_split` (seed = default torch).

**Training:** AdamW, lr 1e-4, 15 epochs, batch 32, `CrossEntropyLoss`. See
`ml-server/train.py`.

**Inference path:**

```
phone → POST /classify {image: b64}  →  server: preprocess → MobileNetV3
                                     →  softmax → top-3
                                     →  binary collapse on `ticks`
                                     →  ScanResult (Trace)
```

**Grad-CAM:** the saliency overlay shown on the Scan tab is computed via
`/explain`, which back-propagates the predicted class score through the last
conv block (`features[-1]`), channel-wise averages the gradients, weights
the activations, ReLUs, upsamples to 224×224, and blends with a red-yellow
LUT. ~150 ms on CPU.

**Privacy:** the photo is sent **only** when `ML_SERVER_URL` is non-null in
`lib/bite-scanner.ts`. With the default `null`, the photo never leaves the
device — the Scan tab routes the user to the on-device questionnaire
(`analyzeBite`).

**Limitations:**
- The training set's EM-rash examples are heavily skewed toward
  light-skinned subjects. Erythema migrans presents differently on darker
  skin. Before any real-world deployment the model should be retrained on a
  demographically-balanced dataset.
- "no_bites" is a catch-all class — any unfamiliar skin patch (pimple,
  scratch, irritation) will be classified as the closest visual match.
- Confidence is the raw softmax. It is not calibrated.

## 2. Fusion head — gradient-boosted risk model

**Architecture:** `sklearn.ensemble.GradientBoostingClassifier`, 80 stages,
max depth 3, learning rate 0.1, multinomial deviance loss. Total size ≈ 98 KB
exported as JSON.

**Why GBM and not a deep model:**
- Calibration is excellent on tabular data with strong prior structure.
  Brier score on the held-out set is **{{see explainability tab}}** — the
  predicted probability matches the empirical frequency closely.
- Interpretable: feature importances drop out for free, and we can do
  cheap single-feature ablation for the per-feature "what the model
  weighted" bars on the Home screen.
- Tiny: 80 stages × 3 classes × shallow trees walks in microseconds in JS,
  ships as a static asset, no native module.

**Training data:** synthetic cohort of 10,000 patients generated by
`ml-server/gen_synthetic_cohort.py`. Class mix: 50% no-Lyme / 35% early-Lyme
/ 15% disseminated-Lyme. Symptom prevalences come from:

- IDSA/AAN/ACR 2020 *Lyme Disease Clinical Practice Guidelines*
- CDC NNDSS Lyme surveillance, 2019–2023
- Steere & Sikand 2003 (early Lyme symptom prevalence)
- Halperin 2015, Forrester et al. 2014 (disseminated Lyme)
- NH DHHS Bureau of Infectious Disease Control surveillance reports

The full prior table is in
[`ml-server/gen_synthetic_cohort.py`](../ml-server/gen_synthetic_cohort.py).

**Feature vector (32 dims):**

```
14 binary symptom flags
 4 one-hot: foundTick      ∈ {no, yes_removed, yes_attached, unsure}
 4 one-hot: rashStatus     ∈ {no, circular, other, unsure}
 4 one-hot: locationRisk   ∈ {nh, other_endemic, non_endemic, unsure}
 4 binary:  outdoorActivity, nearWoods, petsOutdoor, recentFluLike
 1 float:   countyIncidenceRate (cases per 100k, NH counties vs. US baseline)
 1 int:     logCount (days of symptom history, capped at 30)
```

**Held-out metrics** (n=2,000, see `assets/ml-metrics/fusion_metrics.json`):

- Accuracy: 0.9555
- Log loss: 0.1467
- Brier (mean): 0.0249 (per-class: 0.011 / 0.035 / 0.029)
- AUC (macro, OvR): 0.9906

These metrics describe how well the model recovers the **synthetic priors**.
They are **not** an estimate of real-world clinical performance — that would
require an IRB-approved validation study against actual patient outcomes.

**Inference path (pure-TS, on-device):**

```
SymptomChecks + ExposureData + logCount
   → encode (32-element vector matching feature_names)
   → walk all 80 × 3 trees, accumulate per-class scores
   → softmax → [p(no_lyme), p(early_lyme), p(disseminated_lyme)]
   → lymeProbability = p(early) + p(disseminated)
   → score = round(sqrt(lymeProbability) * 100)
   → level = critical (red flags) | high (≥65) | moderate (≥35) | low
```

See [`lib/ml/risk-fusion.ts`](../lib/ml/risk-fusion.ts).

**Per-feature attribution:** single-feature ablation. For each non-zero
feature, we zero it out, re-run the model, and report the resulting drop in
`lymeProbability`. This is not proper SHAP (which is order-invariant and
respects feature interactions) but is computationally cheap and produces
useful UI hints — the bars on the Home screen rank features by absolute
contribution to the headline probability.

## 3. Temporal head — symptom-progression GRU

**Architecture:**

```
x_t ∈ R^14  →  GRU(input=14, hidden=16, layers=1, batch_first=True)  →  Linear(16 → 1)  →  σ
```

1,553 parameters total. The "every step" auxiliary loss supervises every day
of the sequence with the trajectory-level label, which is what gives us a
smoothly-climbing sparkline rather than a confidence that only meaningfully
fires at the last step.

**Training data:** 10,000 synthetic 14-day trajectories
(`ml-server/gen_synthetic_trajectories.py`). 50% positive (Lyme), 25% flu
controls, 25% random-noise controls.

Positive trajectories are drawn from a 3-state Markov chain encoding the
published Lyme progression:

```
flu_like  →  joint_involvement  →  neuro_or_cardiac
```

Stay-in-state probabilities are 0.85/0.85/0.90; per-stage symptom emission
priors mirror the fusion model's class-conditional priors.

**Training:** Adam, lr 3e-3, 15 epochs, batch 128, mixed loss
(0.5 × last-step BCE + 0.5 × every-step BCE).

**Held-out metrics** (n=2,000, see `assets/ml-metrics/temporal_metrics.json`):

- AUC: 1.00
- Brier: 0.0002

The near-perfect score is a **synthetic-data artifact**: the Markov-chain
positives include red-flag symptoms (`facialDroop`, `neckStiffness`,
`heartPalpitations`) at meaningfully higher rates than either control
distribution, which makes the classes nearly linearly separable. On a real
patient dataset where these symptoms are rarer and noisier across all
classes, we would expect substantially lower AUC.

We surface this artifact honestly in the Explainability tab and in the demo
script so a judge isn't misled.

**Inference path (pure-TS, on-device):**

```
SymptomLog[] (newest-first)
   → chronological (oldest first)
   → for each day: encode to 14-element vector
                   → standard GRU update (3 gates, tanh + sigmoid)
                   → linear head + sigmoid
                   → p_t
   → sparkline (Timeline tab)
   → trend = p_last - p_first
```

See [`lib/ml/symptom-progression.ts`](../lib/ml/symptom-progression.ts).

## Heuristic baseline (non-ML)

`lib/risk-engine.ts::calculateRisk` is the original hand-tuned scoring
function: 40 points for symptoms, 30 for exposure, 20 for geography, 10 for
sustained pattern. It remains in the codebase and is shown on the Home screen
as the "Interpretable baseline" alongside the ML score. The intent is that a
viewer can see both numbers agree — the ML model lifts the score in cases
where feature interactions matter, but the heuristic provides a clear,
auditable floor.

This is also the right framing under the Congressional App Challenge's AI
rules: ML supports specific aspects of Trace, but the hand-tuned risk
engine, symptom catalog, NH-data layer, advocacy scripts, PDF generator,
and UI are the student's own non-AI engineering.

## Reproducibility

```bash
cd ml-server
./venv/Scripts/python.exe gen_synthetic_cohort.py --n 10000 --seed 42
./venv/Scripts/python.exe train_risk_fusion.py
./venv/Scripts/python.exe gen_synthetic_trajectories.py --n 10000 --seed 42
./venv/Scripts/python.exe train_temporal.py
```

The three scripts emit JSON artifacts to `ml-server/` and also mirror them
into `../assets/models/` and `../assets/ml-metrics/`. The phone app reads
from `assets/`. Re-running the pipeline regenerates byte-identical outputs
for a given seed.

For the CV head, the dataset is too large to commit; see
[`ml-server/train.py`](../ml-server/train.py) for the Kaggle download
instructions. A pre-trained `model.pt` lives in `ml-server/` and is loaded by
`server.py` on startup.
